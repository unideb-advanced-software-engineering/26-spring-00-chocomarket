# ADR-002: Hybrid Architecture for ChocoMarket

- Status: Active

## Context and Problem Statement

ChocoMarket is a nationwide digital platform within the Zamunda Digital Renaissance (ZDR) program, aimed at consolidating and publishing chocolate manufacturers’ inventory data. The platform must aggregate rapidly changing stock information from heterogeneous sources, operate reliably under imperfect and low-bandwidth network conditions, and remain cost‑efficient as a government-backed initiative.

The key decision is: **Which architectural style best supports reliable inventory ingestion, future growth, maintainability, and economical operation while aligning with the ZDR principles?**

---

## Decision Drivers

- **Scalability** – Must support dozens of manufacturers today, with steady long-term growth and increasing inventory update frequency.
- **Reliability & Data Integrity** – Inventory data should not be lost despite transient failures or network issues.
- **Heterogeneous Integration** – Must ingest data from multiple protocols (FTP, email, HTTP APIs) and formats (CSV, XML, JSON).
- **Connectivity Constraints** – Must function correctly in regions with high latency or unreliable internet access.
- **Cost Efficiency** – Public funds and climate-friendly goals require avoiding over-engineered or wasteful solutions.
- **Maintainability & Extensibility** – New manufacturers, formats, and features (e.g. future purchasing) should be added with minimal disruption.
- **Traceability & Auditability** – Manufacturers must be able to review current and historical inventory data and ingestion attempts.
- **Web Accessibility & Localization** – The platform must provide a multilingual web interface usable worldwide.
- **Future Readiness** – Architecture should allow the later introduction of purchasing and advanced notifications without fundamental redesign.

---

## Considered Options

- **Monolithic Architecture** – Single deployable application handling ingestion, processing, and presentation.
- **Layered (n-tier) Architecture** – Clear separation of presentation, business logic, and data access layers.
- **Service-Based Architecture** – Functionality split into independently deployable services.
- **Event-Driven Architecture (EDA)** – Asynchronous communication via events for ingestion and processing.
- **Hybrid Modular Monolith with Event-Driven Ingestion** – Core system as a modular monolith, with event-driven mechanisms for inventory ingestion and processing.

---

## Decision Outcome

**Chosen option:** *Hybrid Architecture: Modular Monolith combined with Event-Driven Architecture*.

This approach balances reliability, scalability, and cost-efficiency. The modular monolith contains core domain logic (products, inventory aggregation, history, web UI) in a single deployable unit, simplifying deployment and operations. Event-driven mechanisms are used specifically for inventory ingestion and processing, enabling reliable buffering, loose coupling from data sources, and near-real-time updates.

The decision aligns with ZDR goals by avoiding unnecessary architectural complexity while still supporting future growth, diverse integrations, and resilience in challenging network conditions.

---

## Consequences

### Pros

- Reliable ingestion through event-based buffering prevents inventory data loss during transient failures.
- Horizontal scalability of ingestion and processing by adding consumers, without restructuring the core system.
- Loose coupling enables new manufacturers, formats, or ingestion channels to be added with minimal impact.
- Modular monolith reduces operational overhead and infrastructure cost compared to a fully service-based system.
- Event-driven updates support inventory history, price tracking, and stock notification features.

### Cons

- Added operational complexity from managing and monitoring a message broker.
- Asynchronous processing makes testing and debugging more challenging than synchronous flows.
- Event ordering is not guaranteed and must be explicitly handled at the application level (e.g. via timestamps or versioning).
- Schema evolution requires coordination, as multiple consumers may need updates when formats change.

---

## Confirmation

The implementation of the hybrid event-driven, modular-monolith architecture can be confirmed through:

- **Design and Code Reviews** – Ensuring clear module boundaries, appropriate use of events, and adherence to architectural guidelines.
- **Automated Testing** – Unit tests for modules, integration tests for event consumers, and end-to-end tests simulating inventory ingestion scenarios.
- **Monitoring and Observability** – Metrics and logs verifying ingestion latency, event processing success, and system stability.
- **Documentation Reviews** – Validation that ingestion formats, schemas, and processing logic are sufficiently documented, as required by a government program.
- **Fitness Functions** – Automated checks enforcing architectural rules, such as correct module dependencies and event usage patterns.

These measures ensure that the chosen architecture is correctly implemented and continues to meet ChocoMarket’s functional, non-functional, and strategic objectives.
